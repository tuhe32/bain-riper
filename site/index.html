标准工作流

每个 AI 编码任务都遵循严格的闭环流程

1

需求输入

描述需求或上传需求文档（.docx），AI 自动解析并拆分为独立的 OpenSpec change

2

Restate First

AI 用 8 行复述目标 / 边界 / 风险，等你确认后才进入下一步

3

Clarify

主动暴露 3-5 个歧义点，提供选项让你确认，消除边界模糊

4

四件套生成

依次生成 proposal → design → spec → tasks，每步有 Preflight 自检

5

Apply 实施

按 tasks 逐条实现代码，偏差时自动触发 Reverse Sync 修正

6

Verify & Archive

验证实施与 spec 一致后归档，形成可追溯的变更历史

快速开始

三步将 AI 工程化体系引入你的项目

01

CLI 一键初始化

在你的项目根目录执行，自动安装规则、技能、OpenSpec 配置和参考代码

npx bain-riper-cli init --all
02

编辑项目 Profile

修改 openspec/project-profile.md，替换为你的技术栈、包路径和领域约定

# 只需修改这一个文件
openspec/project-profile.md
03

开始 AI 开发

在 AI 编码工具中使用斜杠命令，按标准工作流执行开发任务

/opsx:propose   # 描述需求
/opsx:apply     # 实施代码
/opsx:verify    # 验证一致性

项目宪法 · 10 条底线

AI 在本项目中执行任何任务时必须遵守的最高优先级规则

§1Spec is Truth

spec 是业务行为的唯一真相源

§2Restate First

实施前必须复述目标并获确认

§3多租户完整性

所有业务实体必须含 tenant_id

§4SHALL / MUST

规格书条目必须以强动词开头

§5禁止重复造轮子

新增功能前先检索现有实现

§6禁止猜测 API

必须确认 API 存在且版本兼容

§7标准分层目录

所有 Java 类落在规定目录

§8参考代码对齐

同类功能对齐参考模板风格

§9禁止独立编译

不得单独列出编译验证步骤

§10自检物理可见

进入工作流前展示自检清单
